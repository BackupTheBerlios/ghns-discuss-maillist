Hi!

With the first people being subscribed, the talk can begin. There are many 
areas which were previously touched in private mails and so it'd be a good 
idea to bring on the list of topics again but not to discuss everything at 
once :)

What were the points... let's see:
- the Desktop Exchange Service (DXS) would need a WSDL which specifies which 
parameters it expects for uploads, downloads and other actions (voting could 
be handled too by it couldn't it?)
- common client-side actions, like post-installation scripts, decompression, 
signature verification and "fake installations" (that is, if a distro package 
is upgraded, the download dialog should be able to know if an item has been 
installed already)
- wallpaper-specific options (resolution) -> generic options
- derivates or alternatives
- an icon... waiting for artists
- maybe differential downloads or separate lists for most popular/most 
downloaded/latest
- maybe a way to browse all creative works from an author

Oh and attached you'll find my experiments with SOAP::Lite. The script 
wsclient.pl expects a .meta file as generated for current GHNS uploads, and 
the other two must sit around in the cgi-bin directory and wait for input. No 
real validation is currently being done and authentication is also missing.

For the web pages and the spec files, there are SVN modules. See the BerliOS 
project page for access details.

Josef
