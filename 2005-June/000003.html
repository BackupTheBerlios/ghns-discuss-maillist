Am Donnerstag, 9. Juni 2005 12:11 schrieb Michael Gebhart:
> Well, the first thing we need is a list of the different stuff and a
> description how it has to be installed. So every implementation of GHNS
> can take a look on it and see, what kind of stuff can be downloaded and
> write its own installation procedure. Or did you plan to write some
> basic console scripts to install stuff and that the clients only use
> these scripts?

I planned to have some shell scripts in the end but only to see that it works 
and to make some power users happy :-)
It is true that each implementation expects an own installation location. Even 
in some cases where it is stupid, e.g. the wallpaper, such an intrinsic piece 
of data, and yet so difficult to manage.
So the above is not high priority I'd say.

> - wallpaper-specific options (resolution) -> generic options
>
> With DXS you simply can get a whole datastructure with a list of
> wallpaper specific options.

OK.

> - maybe differential downloads or separate lists for most popular/most
> downloaded/latest
>
> Yeah. Especially with SOAP we can have functions like:
> get_most_popular_theme(start, end) which returns the most popular
> themes, beginning from start to end. Or similar to this:
> get_most_downloaded_theme. Or we simply have functions like:
> get_theme(start, end, sort). So the client has the possibilty to get all
> the themes, it wants to display.

I also think that with SOAP it's possible to reduce traffic by only getting 
the data one really needs. The art.gnome.org XML file is > 6000 lines long.

> - maybe a way to browse all creative works from an author
>
> Yup. I also have some points:
>
> - user registration (if allowed by the administrator)
>
> The client should be able to register some new user accounts for writing
> comments, or uploading stuff.
>
> - read/write comments
>
> A user should be able to read and write comments on the client side. A
> registration should be required.

So there would be a start for WSDL messages already:
* AuthenticationRequest
* ListRequest
* DownloadRequest
* Comment or RateRequest
... and the answers to all of these.

I also had a few ideas. One is that I read about volatile.debian.net and how 
they keep e.g. spamassassin signatures up to date. This is not exactly a 
desktop topic but who knows what kind of data might be next.

The second is just a small improvement over what is currently used:
In addition to the "category" entry (or "type" in our case), a i18n'd human 
readable string should be introduced. For instance, where it now reads 
"freeciv/level", it would then say "Freeciv levels". The 
"kdeedu/vocabularies" category would be "Vocabulary files". And so on.

Josef
