Hi,

> 
> I planned to have some shell scripts in the end but only to see that it works 
> and to make some power users happy :-)
> It is true that each implementation expects an own installation location. Even 
> in some cases where it is stupid, e.g. the wallpaper, such an intrinsic piece 
> of data, and yet so difficult to manage.
> So the above is not high priority I'd say.

Sometimes it is not only done by copying the data to the target
location. For example the GNOME wallpapers has to be added in a xml
file. The scripts should also be able to do this.

> I also think that with SOAP it's possible to reduce traffic by only getting 
> the data one really needs. The art.gnome.org XML file is > 6000 lines long.

Yeah right. This is also an important fact. 

> So there would be a start for WSDL messages already:
> * AuthenticationRequest
> * ListRequest
> * DownloadRequest
> * Comment or RateRequest
> ... and the answers to all of these.

Right. It is also possible to use the HTTP Authentication I heard. But I
am not sure, if this is the right way for authentication.

> I also had a few ideas. One is that I read about volatile.debian.net and how 
> they keep e.g. spamassassin signatures up to date. This is not exactly a 
> desktop topic but who knows what kind of data might be next.

Maybe we need signatures for our desktop data too. E.g. some data (gtk
theme engines) need to be compiled. Eventually we also have other kind
of data that could need a signature to be sure, that the data is checked
and secure. 

> The second is just a small improvement over what is currently used:
> In addition to the "category" entry (or "type" in our case), a i18n'd human 
> readable string should be introduced. For instance, where it now reads 
> "freeciv/level", it would then say "Freeciv levels". The 
> "kdeedu/vocabularies" category would be "Vocabulary files". And so on.

yeah this is nice. But why in addition to the category? Do we need both?
The type freeciv/level etc. should be enough to describe the category.
